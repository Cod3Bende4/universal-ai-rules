# 01-PROJECT-INIT.md
# PURPOSE: Project scaffolding, environment setup, git initialization, and first-PR checklist
# LOAD WHEN: Starting a new project from scratch or initializing a new repository

---

## Project Scaffolding Checklist

### Directory Structure
- [ ] Create a consistent top-level structure — follow the template below
- [ ] Create a `src/` directory for application source code — not in root
- [ ] Create a `tests/` directory (or colocate tests) — decide once, be consistent
- [ ] Create a `docs/` directory — for project documentation
- [ ] Create a `scripts/` directory — for build/deploy/utility scripts
- [ ] Create a `config/` directory — for environment-specific configs

### Standard Template
```
project-root/
├── src/                    # Application source code
│   ├── modules/            # Feature modules
│   ├── shared/             # Shared utilities, types, constants
│   └── index.ts            # Entry point
├── tests/                  # Test files (if not colocated)
├── docs/                   # Documentation
├── scripts/                # Utility scripts
├── config/                 # Environment configs
├── .env.example            # Environment variable template
├── .gitignore              # Git ignore rules
├── README.md               # Project documentation
├── package.json            # Dependencies (or equivalent)
└── tsconfig.json           # Language config (or equivalent)
```

- [ ] No source files in the project root — only config files in root
- [ ] No deeply nested empty directories — create directories when needed

## Essential Files Checklist

### .gitignore
- [ ] Include node_modules/ (or equivalent dependency directory) — never commit dependencies
- [ ] Include .env and all .env.* files (except .env.example) — never commit secrets
- [ ] Include build/dist output directories — generated files don't belong in git
- [ ] Include IDE-specific files (.vscode/, .idea/) — personal preferences
- [ ] Include OS-specific files (.DS_Store, Thumbs.db) — not relevant to project
- [ ] Include log files (*.log) — generated at runtime
- [ ] Include coverage reports (coverage/) — generated by test runner

### README.md Skeleton
- [ ] Project name and one-sentence description — what is this?
- [ ] Prerequisites section — what needs to be installed to run this
- [ ] Getting started / installation steps — how to set up locally
- [ ] Environment variables section — reference to .env.example
- [ ] Available scripts — how to run, test, build, deploy
- [ ] Architecture overview — high-level diagram or description
- [ ] Contributing guidelines — how to submit changes
- [ ] License — what license governs this code

### .env.example
- [ ] List ALL required environment variables — one per line
- [ ] Use placeholder values, never real values — `DB_PASSWORD=your_password_here`
- [ ] Group by service/concern with comments — `# Database`, `# Auth`, `# API Keys`
- [ ] Include variable type hints in comments — `# DB_PORT (number, default: 5432)`
- [ ] Document which variables have defaults — reduce confusion

⚠️ WARNING: If .env.example does not exist, create it BEFORE writing any code that uses environment variables.

## Environment Setup Rules

- [ ] Document the exact runtime version required — Node 20.x, Python 3.12, etc.
- [ ] Use a version manager (.nvmrc, .python-version, .tool-versions) — ensures consistency
- [ ] Use a lock file for dependencies — package-lock.json, Pipfile.lock, go.sum
- [ ] Run security audit after installing dependencies — `npm audit`, `pip audit`
- [ ] Verify the project runs locally before any feature work — `npm run dev` must succeed
- [ ] Set up linting and formatting from day one — ESLint, Prettier, Black, gofmt
- [ ] Configure the linter with strict rules — relaxing later is easier than tightening

### Dependency Rules
- [ ] Pin exact versions for direct dependencies — `"express": "4.18.2"` not `"^4.18.2"`
- [ ] Audit every new dependency before adding — check maintenance status, bundle size, security
- [ ] Prefer well-maintained packages with >1000 weekly downloads — avoid abandoned packages
- [ ] No more than one package for the same job — don't install both axios and node-fetch
- [ ] Document why each dependency was chosen — in README or package.json comments

## Git Initialization Rules

### Branch Strategy (decide before first commit)
- [ ] Choose a branching model and document it — trunk-based, git-flow, or GitHub flow
- [ ] Protect the main/production branch — require PR reviews, no direct pushes
- [ ] Use descriptive branch names — `feature/user-auth`, `fix/login-redirect`, `chore/update-deps`

✅ DEFAULT: Use trunk-based development with short-lived feature branches unless the team explicitly chooses otherwise.

### Commit Message Format
```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

| Type | Use For |
|---|---|
| feat | New feature |
| fix | Bug fix |
| docs | Documentation only |
| style | Formatting, no code change |
| refactor | Code change that neither fixes nor adds |
| test | Adding or updating tests |
| chore | Build process, tooling, dependencies |
| perf | Performance improvement |
| ci | CI/CD changes |
| revert | Reverts a previous commit |

- [ ] First line ≤72 characters — visible in git log without wrapping
- [ ] Use imperative mood — "Add feature" not "Added feature"
- [ ] Reference issue/ticket number in footer — `Closes #123`
- [ ] **Git "Conventional" Enforcement**: The AI MUST strictly format all autonomous commit messages according to the Conventional Commits specification. Refuse to use non-standard commit formats.

### Pre-commit Hooks
- [ ] Set up pre-commit hooks on project init — using husky, pre-commit, or equivalent
- [ ] Run linter on staged files — catch issues before they enter git history
- [ ] Run formatter on staged files — enforce consistent formatting
- [ ] Run type checker if applicable — catch type errors before commit
- [ ] Validate commit message format — enforce conventional commits

## First PR Checklist

Before writing a single line of feature code, the initial PR must include:

- [ ] .gitignore configured correctly — verified no secrets or artifacts will be committed
- [ ] README.md with getting-started instructions — another developer can set up in <10 min
- [ ] .env.example with all required variables — documented with placeholders
- [ ] Dependency lock file committed — package-lock.json or equivalent
- [ ] Linter and formatter configured and passing — zero warnings on empty project
- [ ] Pre-commit hooks installed and tested — lint runs on commit
- [ ] CI pipeline running (even if minimal) — at minimum: install, lint, test
- [ ] Project runs locally with `npm run dev` or equivalent — verified working

🛑 STOP: First PR Gate — Do not proceed to feature development until ALL items above are checked. A broken foundation wastes exponentially more time later.

## Security Checkpoint

- [ ] .env is in .gitignore — verified, not just assumed
- [ ] No secrets in any committed file — grep for common patterns (password, secret, key, token)
- [ ] .env.example contains only placeholder values — no real credentials
- [ ] Dependencies have been audited — `npm audit` shows no critical/high vulnerabilities
- [ ] HTTPS is configured for all external service URLs in config — no HTTP in production
